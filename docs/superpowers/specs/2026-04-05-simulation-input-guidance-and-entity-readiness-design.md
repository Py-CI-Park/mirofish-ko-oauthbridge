# 시뮬레이션 입력 가이드와 엔티티 준비도 진단 개선 설계

## 배경
- 현재 Step2 시뮬레이션 준비 단계는 그래프에서 시뮬레이션용 엔티티를 읽어와 Agent 프로필과 시뮬레이션 설정을 생성한다.
- 실제 운영 사례에서 그래프 노드는 다수 존재했지만, 시뮬레이션 준비 필터 결과는 `0개`였고 준비 단계는 실패했다.
- 동시에 프론트엔드는 실패를 명확히 보여주지 못하고 `config/realtime` 폴링을 계속해, 사용자에게 “계속 진행 중”처럼 보이는 문제가 있었다.
- 원인 분석 결과, 이 현상은 단일 버그가 아니라 다음 요소가 겹쳐 발생했다.
  - 그래프 노드가 시뮬레이션용 행위자보다 기술 변수/지표 중심으로 생성됨
  - Zep 노드의 `labels`가 비어 있어 labels-only 필터가 모두 탈락시킴
  - 사용자가 어떤 첨부파일/프롬프트가 이 프로그램에 적합한지 알기 어려움
  - 실패 이후 UI 상태 전환이 부정확함

## 문제 정의

### 핵심 문제
사용자가 올린 현실 단서와 시뮬레이션 프롬프트가 사회 시뮬레이션용 입력으로 적합하지 않아도, 시스템이 사전에 충분히 안내하거나 차단하지 못한다.

### 파생 문제
1. Step1에서 그래프가 만들어져도 Step2가 요구하는 행위자형 엔티티가 부족할 수 있다.
2. 입력 부적합 상황을 사용자가 이해하기 어렵다.
3. Step2 실패가 명확히 드러나지 않아 “진행 중인지 멈춘 건지” 판단하기 어렵다.
4. 좋은 입력 예시가 부족해 사용자가 시행착오를 반복하게 된다.

## 목표
1. 사용자가 **좋은 첨부파일 + 좋은 시뮬레이션 프롬프트** 조합을 쉽게 이해하도록 한다.
2. 시뮬레이션 준비 실패 시 원인을 명확하고 구조적으로 보여준다.
3. 필요할 때만 사용자가 **필터 완화 재시도**를 선택할 수 있게 한다.
4. 부적합한 입력으로 인해 Step2가 실패하는 빈도를 줄인다.

## 비목표
- 이번 설계는 graph build 자체의 대규모 재설계나 ontology 생성 모델 교체를 포함하지 않는다.
- 자동으로 모든 기술 문서를 사회 시뮬레이션용 행위자 그래프로 변환하는 것을 목표로 하지 않는다.
- Step2를 무조건 성공시키는 품질 저하형 fallback을 기본값으로 두지 않는다.

## 사용자 경험 원칙
- 기본 동작은 **명확한 실패**다.
- 실패 시에는 **진단 정보**를 함께 제공한다.
- 사용자가 원할 경우에만 **통제된 필터 완화 재시도**를 제공한다.
- 안내 문구는 실전형 템플릿을 먼저 보여주고, 왜 좋은지/나쁜지 설명을 덧붙이는 혼합형으로 작성한다.

## 사용자 시나리오

### 시나리오 A: 좋은 현실 단서를 업로드한 사용자
- 기사, 커뮤니티 대화, 이해관계자 소개, 사건 타임라인 등이 포함된 파일을 첨부한다.
- UI 안내에서 추천 입력 형식을 확인한다.
- Step1/Step2가 정상적으로 이어지고 시뮬레이션 준비 성공률이 올라간다.

### 시나리오 B: 기술 문서를 업로드한 사용자
- 전략 설명서, 지표 정의, 함수 목록, 변수 설명서 등을 첨부한다.
- 업로드 후 또는 Step2 시작 전 “이 입력은 사회 시뮬레이션에 부적합할 수 있음” 경고를 본다.
- 계속 진행 시 Step2가 실패하더라도 원인과 다음 조치를 명확히 이해한다.

### 시나리오 C: 그래프가 이미 만들어졌지만 엔티티가 0개인 사용자
- Step2에서 명확한 실패 상태와 함께 아래 정보를 본다.
  - 총 노드 수
  - 매칭 엔티티 수
  - 라벨 유무
  - 적용한 필터 기준
  - 추천 조치
- 원할 경우 “필터 완화 재시도”를 선택한다.

## 제안 범위

### 1. UI 안내 개선
Step1/Step2 화면에 짧은 안내 블록을 추가한다.

#### 핵심 메시지
- **좋은 첨부파일 + 좋은 시뮬레이션 프롬프트를 함께 써야 품질이 좋아집니다.**
- 현실 단서는 사람, 조직, 커뮤니티, 이해관계자, 갈등, 사건, 발화가 드러나는 자료가 적합하다.
- 기술 문서, 전략 설명서, 지표 목록만으로는 사회 시뮬레이션용 엔티티가 잘 만들어지지 않을 수 있다.

#### UI 요약 예시
**좋은 현실 단서**
- 기사 + 댓글 + 반응 모음
- 커뮤니티 게시글 스레드
- 이해관계자 소개 문서
- 갈등 사건 타임라인
- 여러 집단의 입장 정리 자료

**부적합 가능성이 큰 입력**
- 지표/함수 정의서
- 변수 목록
- 전략 공식 설명
- 기술 구현 문서만 있는 파일

#### UI 액션
- `좋은 첨부파일 예시 보기`
- `좋은 시뮬레이션 프롬프트 예시 보기`
- `부적합한 입력 예시 보기`

### 2. 문서 가이드 추가/강화
문서에는 자세한 좋은/나쁜 사례와 프롬프트 템플릿을 넣는다.

#### 추천 위치
- `README.ko.md`: 짧은 요약
- `docs/usage-guide-ko.md`: 상세 사용 흐름
- 신규 또는 확장 문서: 시뮬레이션 입력 가이드

#### 문서 구성
1. 어떤 첨부파일이 좋은 현실 단서인가
2. 어떤 첨부파일이 부적합한가
3. 좋은 시뮬레이션 프롬프트 예시
4. 나쁜 프롬프트 예시
5. 왜 이런 차이가 결과 품질에 영향을 주는가

### 3. 사전 경고(입력 품질 경고)
텍스트 추출 후 또는 Step2 시작 전, 입력이 부적합해 보이면 경고한다.

#### 경고 신호 예시
- 사람/조직/커뮤니티 관련 명칭 비중이 매우 낮음
- 기술 용어/지표/함수명 비중이 매우 높음
- 발화/갈등/상호작용/입장 정보가 거의 없음
- 추출된 그래프 노드가 대부분 변수/지표/연산 개념으로 보임

#### 경고 UX
- 치명적 차단이 아니라 우선 경고
- 사용자는
  - 계속 진행
  - 좋은 예시 보기
  - 입력 교체
중 하나를 선택할 수 있다.

### 4. 실패 시 진단 정보 노출
Step2 실패 화면 또는 고급 영역에 다음을 노출한다.

#### 진단 정보
- 그래프 총 노드 수
- 필터 통과 엔티티 수
- 발견된 엔티티 타입 수
- 노드 라벨 사용 가능 여부
- 현재 필터 기준 요약
- 실패 이유
- 추천 다음 조치

#### 예시 메시지
> 그래프에서 총 180개 노드를 읽었지만, 시뮬레이션 가능한 엔티티는 0개였습니다.  
> 현재 그래프는 기술 변수/지표 중심이거나, 엔티티 라벨이 충분하지 않을 수 있습니다.  
> 좋은 입력 예시를 확인하거나, 필터 완화 재시도를 선택해 주세요.

### 5. 필터 완화 재시도
기본은 엄격 실패지만, 사용자가 원할 때만 완화 재시도를 허용한다.

#### 기본 정책
- **기본 모드:** strict
- **선택 모드:** relaxed

#### strict 모드
- labels 기반 정식 엔티티 타입 우선
- actor/non-actor 기준을 통과한 엔티티만 허용
- 0개면 실패

#### relaxed 모드
- `labels + summary + attributes`를 함께 사용
- 라벨이 비어도 의미 기반으로 actor 후보를 추정
- 단, 아래는 기본 제외
  - 순수 지표명
  - 함수명
  - 계산식
  - 변수 컬렉션/기술 항목

#### relaxed 모드 후보 판별 방향
- 이름과 summary에 사람/조직/커뮤니티/계정/전문가/기관 등의 역할 단서가 있는가
- 사회적 발화나 행동 단서가 있는가
- “지표”, “변수”, “수식”, “조건”, “함수”, “indicator”, “ratio”, “strategy rule” 류면 제외

### 6. UI 상태 처리 보정
실패를 완료처럼 보이게 하는 흐름을 막는다.

#### 기대 동작
- Step2 prepare가 실패하면 UI는 즉시 실패 상태로 전환
- `config/realtime` 무한 폴링 중단
- 실패 카드와 진단/재시도 옵션 표시

#### 핵심 보정 포인트
- prepare task 상태와 simulation state의 실패 여부를 함께 본다
- `config_generated=false`만으로 “생성 중”으로 간주하지 않는다
- simulation state가 `failed`면 config polling을 중단한다

## 아키텍처 관점 설계

### 백엔드
- 엔티티 판별 결과를 더 풍부하게 반환
  - total_nodes
  - filtered_entities
  - labels_present_ratio
  - actor_candidate_count
  - failure_reason
- strict / relaxed 모드를 구분
- 실패 결과를 task 완료와 구별 가능한 형태로 유지

### 프론트엔드
- Step2 안내 블록 추가
- 실패 시 고급 진단 패널 표시
- `필터 완화 재시도` 액션 제공
- `좋은 예시 보기` 링크 제공

### 문서
- 빠른 시작 가이드에는 한눈에 보는 요약
- 상세 문서에는 첨부파일/프롬프트 예시와 이유 설명

## 접근안 비교

### 접근안 A — 문서만 추가
- 장점: 구현이 가장 간단함
- 단점: 실제 입력 시점에서 사용자가 놓치기 쉽다

### 접근안 B — UI만 추가
- 장점: 즉시 눈에 띈다
- 단점: 예시와 이유를 충분히 설명하기 어렵다

### 접근안 C — UI 요약 + 문서 상세 + 실패 시 연결 + 사전 경고
- 장점: 예방, 학습, 실패 복구가 한 흐름으로 이어진다
- 단점: 구현 범위가 상대적으로 크다

### 추천 접근안
**접근안 C**

## 구현 우선순위
1. Step2 실패를 실패로 보이게 하는 상태 처리 수정
2. 실패 화면 진단 정보와 재시도 옵션 추가
3. UI 요약 안내 추가
4. 문서 예시/템플릿 추가
5. 입력 품질 사전 경고 추가
6. relaxed 재시도 규칙 정교화

## 검증 계획

### 기능 검증
- 좋은 현실 단서 업로드 시 Step2 성공 흐름 확인
- 부적합한 입력 업로드 시 경고 또는 명확한 실패 확인
- 실패 후 config polling이 멈추는지 확인
- relaxed 재시도 시 후보 추정이 동작하는지 확인

### 회귀 검증
- 기존 정상 입력은 strict 모드에서 계속 성공해야 한다
- 실패 화면에서 잘못된 성공 배지가 나타나지 않아야 한다
- 문서 링크와 예시가 실제 경로와 일치해야 한다

## 리스크
- relaxed 모드가 과도하면 기술 변수/지표를 사람/조직처럼 오분류할 수 있다
- 가이드 문구가 길어지면 사용자에게 부담이 될 수 있다
- 도메인마다 “좋은 현실 단서” 정의가 조금씩 달라질 수 있다

## 완화 전략
- strict를 기본값으로 유지한다
- relaxed는 사용자가 명시적으로 선택할 때만 활성화한다
- UI는 짧은 요약, 문서는 자세한 설명으로 역할을 분리한다
- actor/non-actor 판별 규칙은 테스트 케이스와 함께 점진적으로 보정한다

## 성공 기준
- 사용자가 입력 형식을 이해하기 쉬워진다
- Step2 실패 시 원인 파악 시간이 줄어든다
- “멈춘 것 같은데 실제로는 실패한 상태” 오해가 줄어든다
- 부적합한 입력으로 인한 prepare 실패율이 감소한다
